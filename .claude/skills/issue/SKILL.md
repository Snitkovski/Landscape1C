# Отработка GitHub-ищью

Ищью приходят из форм на сайте: метка **«правка»** (шаблон fix-card — исправить карточку) и **«предложение»** (шаблон add-tool — новый инструмент).

## 1. Прочитай ищью

```bash
gh issue list --state open
gh issue view <N> --json title,body,author,labels,comments
```

## 2а. Метка «правка» — исправление карточки

1. Найди карточку в `app/data.js`, проверь факты из ищью (ссылки — curl-ом, живость/содержимое).
2. Внеси правку: канон-форма файла (см. CLAUDE.md), `updated` карточки = сегодня. Попутные улучшения той же карточки уместны (например, ссылка «с чего начать» с человеческой подписью вместо голого URL), чужие карточки не трогай.
3. `node scripts/validate.js && node scripts/sitegen.js && node scripts/cachebust.js`.
4. Коммит в стиле `Данные: <что сделано> (#N)`, push в `main` (автодеплой на landscape1c.ru).
5. Закрой ищью комментарием: что сделали, чем дополнили сверх просьбы, хеш коммита. Тон — благодарный и конкретный.

## 2б. Метка «предложение» — новый инструмент

Сначала оцени по критериям `CONTRIBUTING.md` («Что попадает в ландшафт»): реальное применение в 1С-практике / живость / это инструмент. Собери факты сам: установки в маркетплейсах, звезды и свежесть репозитория, следы применения за пределами автора (Инфостарт, форумы, доклады).

**Прецеденты отказов** (держи формулировки консистентными):

- **#23** — свежий репозиторий (~месяц, 26 звезд, 0 форков), используется только автором → «пока не добавляем: хотим видеть признаки применения в практике за пределами автора; вернемся, когда появятся».
- **#15** — качественный, но вне границ проекта (прикладной low-code конструктор, а не инженерная практика) → «причина не в качестве, а в границах проекта».
- **#14** — нет прозрачности: не нашли лицензию, автора/организацию, планы развития → «вопрос устойчивости и прозрачности проекта».

Если инструмент проходит — добавляй скилом `add-tool` и закрывай ищью с итогом разметки и хешем коммита. Если нет — вежливый отказ по образцу прецедентов: поблагодарить, объяснить критерий, оставить дверь открытой («аргументированные возражения приветствуются»). Спорные случаи — покажи пользователю факты и рекомендацию, решение за ним.

## Общее

- Ответы в ищью — по-русски, без буквы «ё», без канцелярита.
- Решение об отказе — только после разведки фактов, не по названию.
- Один коммит на одно ищью; номер ищью в сообщении коммита (`#N`).
