# ТЗ: Интерактивный ландшафт технологий 1С

## 1. Цель и видение

Превратить линейный каталог [StackTechnologies1C](https://github.com/Oxotka/StackTechnologies1C) (один большой README) в **интерактивную карту** экосистемы 1С.

Главное отличие от обычного списка — **навигация и мнение**:
- фильтры по роли, контексту работы, зрелости, лицензии, происхождению и доступности на рынке РФ;
- карточки с описанием, ответом «зачем» и блоком «с чего начать» (сильная сторона исходного каталога, её сохраняем);
- связи между технологиями (аналоги и зависимости) и отдельная страница графа связей;
- для open-source — автоматические метрики (звёзды, последний коммит, лицензия) — *планируется*.

## 3. Функциональные требования

- **Карточки** технологий: логотип, краткое описание, метки.
- **Категории и блоки** — отправная структура из README дополнена и уточнена в ходе наполнения. Сейчас 19 категорий (поле `categories` в `app/data.js`) сгруппированы в 6 тематических блоков (поле `blocks`):
  - **Проектирование**: Прототипирование · Моделирование и документация
  - **Разработка**: Разработка · Плагины и библиотеки · ИИ-ассистенты · Системы хранения версий · CI/CD
  - **Качество**: Автоматизированное тестирование · Статический анализ кода · Код-ревью
  - **Инфраструктура и эксплуатация**: Администрирование и инфраструктура · СУБД · Мониторинг · Производительность и нагрузочное тестирование
  - **Данные и интеграции**: Интеграции · API и веб-сервисы · BI-системы
  - **Управление и компетенции**: Задачи · Сертификация
- **Фильтры и поиск** (две независимые оси — функция и контекст — плюс три скалярные метки и доступность):
  - **функциональная роль** (*что делаешь*): разработчик / администратор / аналитик / тестировщик (роль «администратор» включает devops/эксплуатацию);
  - **контекст работы** (*где/как работаешь*): франчайзи / инхаус / продукты / проекты (бывший «фриланс» влит во «франчайзи»);
  - **зрелость**: `базовое` / `продвинутое` / `нишевое` (уровень `устаревает` убрали — см. METHODOLOGY.md);
  - **лицензия**: open-source / бесплатное / проприетарное;
  - **происхождение**: отечественное / зарубежное (импортозамещение);
  - **доступность** на рынке РФ: доступен / ограничен (отдельно от происхождения).
- **Стартовые наборы** по контексту работы изначально планировались отдельными курируемыми подборками-кнопками; в текущей версии от kit-кнопок отказались — их роль выполняет фильтр по контексту (выбрал «инхаус» → видишь его инструменты).
- **Деталь карточки** (модалка): описание, «зачем», «с чего начать», ссылки; для OSS — авто-метрики.
- **Виды отображения**: ландшафт (карточки) + граф связей; опционально радар-вид по зрелости.
- **Embed** карточек на внешние сайты — *возможная фича на будущее*.

## 4. Модель данных

Данные живут в `app/data.js` — глобальный объект `window.LANDSCAPE` (`categories`, `blocks`, `axes`, `items`). Одна карточка из `items`:

```js
{
  name: "1C: Platform Tools",
  category: "Разработка",          // одна из categories
  subcategory: null,
  description: "Расширение VSCode: инструменты разработки 1С в одной панели",
  why: "Не переключаться между утилитами при разработке около-1С проектов",
  logo: "1c-platform-tools.svg",   // файл в app/logos/; null → заглушка «1С»
  homepage: "https://marketplace.visualstudio.com/items?itemName=yellow-hammer.1c-platform-tools",
  repo: "https://github.com/yellow-hammer/vscode-1c-platform-tools", // → авто-метрики (план)
  start: ["Marketplace: …", "Репозиторий: …"],   // блок «с чего начать»
  maturity: "продвинутое",         // базовое | продвинутое | нишевое
  roles: ["разработчик", "администратор"],        // функциональная роль (массив)
  contexts: ["инхаус", "продукты"],               // контекст: франчайзи|инхаус|продукты|проекты
  origin: "отечественное",         // отечественное | зарубежное
  license: "open-source",          // open-source | бесплатное | проприетарное
  availability: "доступен",        // доступен | ограничен (доступность на рынке РФ)
  analogs: ["…"],                  // взаимозаменяемые (симметрично)
  depends: ["…"],                  // «зависит от» (направленно)
  aliases: ["гит", "git hub"]      // скрытые поисковые синонимы (рус. транслитерации и пр.)
}
```

**Инвариант значений:** строковые метки в карточке должны точно совпадать со значениями в `axes.<ось>.values` — иначе фильтр не сработает. Целостность проверяет `scripts/validate.js`.

## 5. Архитектура

**Свой статический фронтенд** (HTML/CSS/JS без сборки и зависимостей), данные — `app/data.js`, деплой статикой на GitHub Pages. К этому решению пришли, отвергнув готовый движок (см. п.10–11).

## 6. Контекст работы: стартовые наборы

Контекст работы 1С-разработчика заметно меняет набор инструментов. Реализовано меткой-осью `contexts` на элементах (отдельных курируемых kit-подборок в текущей версии нет — см. п.3):

| Контекст | Что в фокусе | Характерные инструменты |
|---|---|---|
| **Франчайзи** (+ бывший фриланс) | много клиентских баз, типовые, обновления, быстрые доработки; минимум инфраструктуры | Конфигуратор, ИТС, расширения, многобазная обработка, обмены, портативные инструменты, Infostart Toolkit |
| **Инхаус** | одна компания, глубокая поддержка и интеграции | EDT+git, мониторинг (Zabbix/Prometheus), брокеры, S3, кластер |
| **Продукты** (тиражные решения) | релизы, совместимость версий, качество | git, CI/CD (Jenkins/GitLab), Vanessa/YAxUnit, SonarQube/BSL LS, OPM, публикация на Инфостарт |
| **Проекты** (внедрение) | методология, крупные внедрения, нагрузка | СППР, EnterpriseData/Шина, 1С:КИП/ЦКК, нагрузочное тестирование, документирование |

Это не жёсткие границы, а ориентир: один инструмент может попадать в несколько контекстов.

## 7. Миграция данных из README

README исходного каталога структурирован единообразно:
`заголовок → описание → «Зачем нужно» → «С чего начать» → ссылки`.

Для первичного переноса секций использовались одноразовые Python-скрипты, но дальше всё доводилось и размечалось вручную (135 карточек на текущий момент), метки и логотипы проставлены руками. Миграция завершена, скрипты больше не нужны и удалены из репозитория; дальнейшее наполнение идёт через редактор (п.16).

**Гранулярность: одна карточка = один инструмент (атомарность).** Ландшафт мельче нарезан, чем README:
- README местами группирует несколько инструментов в один «зонтичный» раздел (например, «Брокеры сообщений (RabbitMQ, Kafka)»; «Мониторинг и аналитика» с Zabbix/Prometheus/ELK/ClickHouse). В ландшафте каждый из них — **отдельная карточка**.
- Пример: `gitConverter` и `gitSync` — две разные карточки, а не одна.
- Следствие: **инструментов заметно больше, чем разделов README**; миграция **не 1:1** — зонтичные разделы дробились на атомарные карточки вручную.

## 9. Принятые решения

- **Шкала зрелости**: своя — `базовое` / `продвинутое` / `нишевое` (уровень `устаревает` ввели, затем убрали — см. METHODOLOGY.md). (Tech Radar — возможно позже.)
- **Язык интерфейса**: только русский.
- **Связь с исходным репо**: проекты независимы. Исходный [StackTechnologies1C](https://github.com/Oxotka/StackTechnologies1C) остаётся как есть; сюда забираем только нужные данные.
- **Рубрика разметки** (`roles/contexts/maturity/...`): пока размечаем «на глаз», единые критерии формализуем позже.
- **MVP-категории**: IDE, CI/CD, Системы хранения версий.

## 10. Почему не готовый движок (landscape2)

Сначала пробовали [landscape2](https://github.com/cncf/landscape2) (движок CNCF Landscape: YAML → статика, авто-метрики GitHub). Прототип показал, что он не закрывает ключевую идею проекта:

- **Кастомные фильтры не поддерживаются.** Набор фильтров фиксирован (категория, зрелость, лицензия, организация, страна + общий `TAG`). Наши две оси (роль + контекст) чисто отфильтровать нельзя; кастомные поля движок принимает, но не использует ни в фильтрах, ни на карточках.
- **Нет локализации UI.** Обвязка интерфейса — на английском, что конфликтует с решением «только русский».

**Итог: от landscape2 отказались**, артефакты разведки (его `landscape.yml`/`settings.yml` и сборку) удалили из репозитория.

## 11. Архитектурное решение

**Делаем свой статический фронтенд.** Две оси фильтров (роль + контекст) + зрелость/происхождение/лицензия/доступность, полностью русский UI, связи и граф. HTML/CSS/JS без сборки и зависимостей, данные — `window.LANDSCAPE` в `app/data.js`. Деплой статикой на GitHub Pages.

## 12. Связи между технологиями (реализовано, включая граф)

Связи превращают каталог в карту и дают навигацию «что дальше».

**Реализовано.** Два типа связей, по образцу схемы из StackTechnologies1C (`img/stacktechnology.png`):
- **Аналоги** — взаимозаменяемые инструменты (`RabbitMQ ↔ Kafka`, `Конфигуратор ↔ EDT ↔ VSCode`, СУБД между собой).
- **Зависимости** — связанные по необходимости (`EDT — git`, `Хранилище — Конфигуратор`, плагины → их хосты, `Kibana/Logstash → Elasticsearch`).

**Модель (фактическая).** У элемента два поля-массива `analogs` и `depends` (имена связанных карточек):
- **`analogs` — симметричны**: прописываются на обоих концах. Группируются по близости («простые» и «сложные» внутри одной категории связываются раздельно, а не все-со-всеми).
- **`depends` — направленные («зависит от»)**: A → B означает «чтобы осмысленно работать с A, нужен и B», и так выстраивается цепочка (IDE → git → Хранилище). Прописываются только на исходном конце.

В детальной модалке (`detail.js`) — блоки «Аналоги»/«Зависимости» с квадратными кнопками (лого + название), клик открывает связанный инструмент. Сейчас связи проставлены примерно у 100 карточек (аналоги) и 60 (зависимости). Целостность связей проверяет `scripts/validate.js`.

**Граф связей (реализован).** Отдельная страница `graph.html` (`graph.js`): force-directed canvas-граф по `analogs`/`depends` с фильтрами по роли и зрелости, тёмной темой и поддержкой тач-жестов на узких экранах. Клик по узлу открывает ту же детальную модалку, что и на главной.

**Отличия от первоначального плана.** Свели 5 предложенных типов к двум (аналоги/зависимости) — этого достаточно для навигации и совпадает с авторской схемой. Сделана направленность зависимостей и полноценный граф. На будущее: редактирование связей в `editor.html`.

## 13. Слой компетенций и грейдов (планируется)

Третий слой, связывающий инструменты с карьерным ростом. Источники — матрицы компетенций (см. справочные ссылки). Идея:

- **Компетенция ↔ технологии** (DevOps/CI-CD → Jenkins/GitLab/Docker; нагрузочное тестирование → КИП/Тест-центр).
- **Компетенция ↔ грейд** (Junior/Middle/Senior/…): что требуется на каждом уровне.
- Вид **«Грейд»**: выбрал уровень роли → видишь нужные компетенции и стоящие за ними технологии («что учить, чтобы вырасти»).

Открытые вопросы по матрицам: расшифровать легенду отметок `v`/`ж`/`Ж`; согласовать расходящиеся версии (листы «Аналитики» vs «Программисты»); встраивать отдельным слоем, не смешивая с фильтрами инструментов.

## 14. Экспертность, контрибьюторы и методология (частично реализовано)

Ценность ландшафта — в мнении (зрелость, рекомендации, связи), а мнению нужна видимая экспертиза. Делаем ставку на коллективную экспертизу уважаемых в сообществе людей, а не одного автора.

**Согласие (обязательно, до публикации).** Фото, имя и контакты любого человека — только с явного разрешения. Ссылаться на публичные профили (Telegram, GitHub, Инфостарт), а не на личные контакты; аватар — только с согласия. Процесс: короткое согласование «как это будет выглядеть» + ссылка на черновик.

**Как показываем экспертность (вводим постепенно):**
1. **Экспертный совет** — *реализован* (`council.html`): страница с кураторами (фото, имя, био, область курирования) и автором проекта. Сейчас 2 куратора (аналитика, разработка) + автор; фото лежат в `app/people/`. Расширяется по мере согласия новых людей.
2. **Контрибьюторы** — открытый список всех участников (как в awesome/CNCF), мотивирует вносить вклад. *Планируется.*
3. **Атрибуция оценок** — «категорию курировал X», «метку зрелости подтвердили …» (приём Thoughtworks Tech Radar). *Планируется.*

**Страница «Методология»** — *реализована* (`methodology.html`): как принимаются решения, как размечены оси, как предложить правку. Прозрачность = доверие. Рабочая версия правил разметки осей (роль, контекст, зрелость, происхождение, лицензия, доступность) и сравнение с другими ландшафтами ведётся в [`METHODOLOGY.md`](METHODOLOGY.md).

**Модель данных:** `experts: [{ name, role, photo, links, areas }]`; опционально `curatedBy` у категорий/элементов. Реализация — статикой.

**Порядок:** прототип, связи, разделы «Экспертный совет» и «Методология» — сделаны. Дальше: согласовывать новых экспертов и открыть список контрибьюторов.

## 15. Популярность у сообщества (планируется)

Помимо экспертного мнения — собирать **реальную распространённость** технологий: насколько они используются на практике. Это второй сигнал ценности (мнение экспертов + факт использования).

**Два подхода:**
1. **Разовый опрос раз в год** *(рекомендуется первым)*. Внешняя форма (Google Forms / Tally / Яндекс Формы) → агрегированные результаты публикуем статикой и показываем на карточках. Плюсы: не нужен бэкенд, устойчивее к накрутке, даёт срез динамики год-к-году (как «State of JS»).
2. **Живые счётчики «Я использую»** на карточке. Плюс: всегда актуально. Минус: требует **бэкенда/хранилища** (Pages — статика, сама не сможет), и сложнее защита от накрутки.

**Нюансы (заранее):**
- **Архитектура.** Живые счётчики ломают чисто-статическую модель — понадобится serverless-функция + БД/KV или сторонний сервис. Опрос этого не требует.
- **Накрутка.** Для опроса — одна отправка/респондент, модерация; для живых счётчиков — антифрод (аутентификация, rate-limit), что заметно сложнее.
- **Приватность.** Собирать обезличенно; согласие на обработку, если будут персональные данные.

**Рекомендация:** начать с ежегодного опроса (низкая инфраструктура, честнее), публиковать срез и динамику. Живые счётчики — позже, только если появится бэкенд.

## 16. Инструмент редактирования и разметки (реализован)

Править данные напрямую в `app/data.js` (бегать по JSON руками) неудобно и чревато ошибками: легко опечататься в значении метки, рассинхронить его с `axes.*.values`, забыть поле. Нужен **визуальный редактор разметки**.

Идея — тот же вид, что и основной ландшафт (сетка карточек), но в **режиме редактирования**:
- менять метки осей (`роль / контекст / зрелость / происхождение / лицензия`) кликами по чипам, а не текстом;
- редактировать `name / description / why / start / связи` в форме карточки;
- **валидация по инварианту значений**: метки выбираются только из допустимых `axes.*.values`, рассинхрон невозможен;
- добавление и удаление карточек.

Вывод — **экспорт обратно в формат `app/data.js`**, чтобы результат можно было закоммитить.

**Реализовано** (`app/editor.html` + `editor.js`): сетка карточек в режиме редактирования, метки осей кликами по чипам с валидацией по `axes.*.values`, добавление/удаление карточек. По кнопке «Сохранить» весь `window.LANDSCAPE` сериализуется обратно тем же 2-пробельным JSON и пишется прямо в `app/data.js` — для записи нужен лёгкий пишущий сервер `scripts/serve.py` (POST `/save`); без него «Сохранить» откатывается на скачивание копии. Дата `updated` проставляется автоматически.
