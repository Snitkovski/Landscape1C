# ТЗ: Интерактивный ландшафт технологий 1С

## 1. Цель и видение

Превратить линейный каталог [StackTechnologies1C](https://github.com/Oxotka/StackTechnologies1C) (один большой README) в **интерактивную карту** экосистемы 1С.

Главное отличие от обычного списка — **навигация и мнение**:
- фильтры по роли, зрелости, лицензии, происхождению;
- карточки с описанием, ответом «зачем» и блоком «с чего начать» (сильная сторона исходного каталога, её сохраняем);
- для open-source — автоматические метрики (звёзды, последний коммит, лицензия).

## 3. Функциональные требования

- **Карточки** технологий: логотип, краткое описание, метки.
- **Категории/подкатегории** — берём готовую структуру из README (Разработка, СУБД, Системы хранения версий, Повышение качества, CI/CD, Интеграции, BI/Мониторинг, Администрирование, Сертификация).
- **Фильтры и поиск** (две независимые оси — функция и контекст):
  - **функциональная роль** (*что делаешь*): разработчик / администратор / DevOps / аналитик / тестировщик;
  - **контекст работы** (*где/как работаешь*): франчайзи / инхаус / продукты / проекты / фриланс;
  - **зрелость**: `базовое` / `продвинутое` / `нишевое` / `устаревает`;
  - **лицензия**: open-source / бесплатное / проприетарное;
  - **происхождение**: отечественное / зарубежное (импортозамещение).
- **Курируемые стартовые наборы** по контексту работы — отдельные подборки-«пути» («с чего начать франчайзи / инхаусу / продуктовику / проектнику / фрилансеру»), как роль-пути на roadmap.sh. Дополняют фильтр-метки, задают рекомендуемый минимум для каждого контекста.
- **Деталь карточки** (модалка): описание, «зачем», «с чего начать», ссылки; для OSS — авто-метрики.
- **Виды отображения**: ландшафт (карточки) + список; опционально радар-вид по зрелости.
- **Embed** карточек на внешние сайты (есть в landscape2 из коробки).

## 4. Модель данных

Один элемент в `landscape.yml` (доменные поля 1С — в блоке `extra`):

```yaml
- item:
    name: "1C: Platform Tools"
    description: "Расширение VSCode: инструменты разработки 1С в одной панели"
    homepage_url: https://marketplace.visualstudio.com/items?itemName=yellow-hammer.1c-platform-tools
    repo_url: https://github.com/yellow-hammer/vscode-1c-platform-tools   # → авто-метрики
    logo: 1c-platform-tools.svg
    extra:
      maturity: продвинутое        # базовое|продвинутое|нишевое|устаревает
      roles: [разработчик, devops] # функциональная роль
      contexts: [инхаус, продукты] # контекст работы: франчайзи|инхаус|продукты|проекты|фриланс
      origin: отечественное        # отечественное|зарубежное
      license_type: open-source    # open-source|бесплатное|проприетарное
      why: "Не переключаться между утилитами при разработке около-1С проектов"
      getting_started:
        - "Marketplace: https://marketplace.visualstudio.com/items?itemName=yellow-hammer.1c-platform-tools"
        - "Репозиторий: https://github.com/yellow-hammer/vscode-1c-platform-tools"
```

## 5. Архитектура

**Вариант A (рекомендуемый): landscape2**
- Движок: `landscape2` (Rust CLI), на выходе — статический сайт.
- Данные: `data/landscape.yml` + `data/settings.yml` + `logos/` (SVG).
- Сборка/деплой: GitHub Actions → GitHub Pages (как Jekyll в исходном репо).

**Вариант B (запасной): свой фронтенд**
- React/Astro + JSON-данные. Берём только если упрёмся в ограничения landscape2 (например, своя логика фильтров/радара).

## 6. Контекст работы: стартовые наборы

Контекст работы 1С-разработчика заметно меняет набор инструментов. Реализуем двумя способами (фильтр-метка `contexts` на элементах + курируемые подборки):

| Контекст | Что в фокусе | Характерные инструменты |
|---|---|---|
| **Франчайзи** | много клиентских баз, типовые, обновления, быстрые доработки | Конфигуратор, ИТС, расширения, многобазная обработка, обмены |
| **Инхаус** | одна компания, глубокая поддержка и интеграции | EDT+git, мониторинг (Zabbix/Prometheus), брокеры, S3, кластер |
| **Продукты** (тиражные решения) | релизы, совместимость версий, качество | git, CI/CD (Jenkins/GitLab), Vanessa/YAxUnit, SonarQube/BSL LS, OPM, публикация на Инфостарт |
| **Проекты** (внедрение) | методология, крупные внедрения, нагрузка | СППР, EnterpriseData/Шина, 1С:КИП/ЦКК, нагрузочное тестирование, документирование |
| **Фриланс** | минимум инфраструктуры, мобильность | портативные инструменты, Infostart Toolkit, расширения, Конфигуратор |

Это не жёсткие границы, а ориентир: один инструмент может попадать в несколько контекстов. Курируемый набор задаёт рекомендуемый минимум «с чего начать» для каждого контекста.

## 7. Миграция данных из README

README исходного каталога структурирован единообразно:
`заголовок → описание → «Зачем нужно» → «С чего начать» → ссылки`.

Это позволяет написать **скрипт-парсер** (OneScript или Python), который вытащит секции в данные — ~80% наполнения автоматически. Дальше вручную проставляются метки (роль / зрелость / происхождение / лицензия) и подбираются логотипы.

**Гранулярность: одна карточка = один инструмент (атомарность).** Ландшафт мельче нарезан, чем README:
- README местами группирует несколько инструментов в один «зонтичный» раздел (например, «Брокеры сообщений (RabbitMQ, Kafka)»; «Мониторинг и аналитика» с Zabbix/Prometheus/ELK/ClickHouse). В ландшафте каждый из них — **отдельная карточка**.
- Пример: `gitConverter` и `gitSync` — две разные карточки, а не одна.
- Следствие: **инструментов будет заметно больше, чем разделов README**; миграция **не 1:1** — зонтичные разделы дробятся, парсер должен это учитывать (либо досоздаём карточки вручную).

## 8. Этапы

1. **MVP**: поднять `landscape2` на категориях **IDE**, **CI/CD**, **Системы хранения версий**, деплой на Pages.
2. **v1**: парсер README → полное наполнение; логотипы; фильтры по роли и зрелости.
3. **v2**: метки происхождения/лицензии, авто-метрики OSS, радар-вид, embed-карточки.

## 9. Принятые решения

- **Шкала зрелости**: своя — `базовое` / `продвинутое` / `нишевое` / `устаревает`. (Tech Radar — возможно позже.)
- **Язык интерфейса**: только русский.
- **Связь с исходным репо**: проекты независимы. Исходный [StackTechnologies1C](https://github.com/Oxotka/StackTechnologies1C) остаётся как есть; сюда забираем только нужные данные.
- **Рубрика разметки** (`roles/contexts/maturity/...`): пока размечаем «на глаз», единые критерии формализуем позже.
- **MVP-категории**: IDE, CI/CD, Системы хранения версий.

## 10. Результат разведки landscape2 (важно)

Проверка движка показала ограничения, влияющие на архитектуру:

- **Кастомные фильтры не поддерживаются.** Набор фильтров фиксирован: категория/подкатегория, зрелость, лицензия, организация, страна и один общий фильтр `TAG`. Наши две оси (`роль` и `контекст работы`) чисто отфильтровать нельзя — максимум одна через `TAG`, вторая только на карточке.
- **Нет локализации UI.** Обвязка интерфейса — на английском; контент/категории можно по-русски. Конфликтует с решением «только русский».
- Логотипы — обязательно SVG. Установка — бинари/Homebrew/Docker/cargo.

**Вывод:** Вариант A (landscape2) не закрывает полностью ключевую идею (две чистые оси + русский UI). Развилка архитектуры — см. ниже, решение не принято.

## 11. Архитектурное решение

**Выбран Вариант B — свой фронтенд.**

Прототип на landscape2 подтвердил ограничения: кастомные поля (`maturity/roles/contexts/...`) движок принимает, но не использует — ни в фильтрах, ни на карточках. Из коробки доступен только фильтр по категории (+ нативные license/maturity/теги при заполнении), UI английский. Две чистые оси (роль + контекст работы) реализовать нельзя.

Поэтому делаем свой статический фронтенд: две оси фильтров + зрелость/происхождение/лицензия, курируемые наборы по контекстам, полностью русский UI. Деплой статикой на GitHub Pages. Данные — JSON (конвертируются из наработанного `data/landscape.yml`).

Артефакты landscape2 (`data/landscape.yml`, `settings.yml`, `logos/`) остаются в репозитории как документированный результат разведки.

## 12. Связи между технологиями (планируется)

Связи превращают каталог в карту и дают навигацию «что дальше». Типы связей:

- **основа для / предшествует** — `git → CI/CD`, `OneScript → vanessa-runner`
- **используется вместе** — `Jenkins + Docker + Allure`
- **альтернатива** — `Хранилище ↔ git`, `Jenkins ↔ GitLab CI`
- **входит в набор / часть** — компоненты 1С:КИП
- **расширяет** — плагины к EDT/Конфигуратору

Модель: у элемента `related: [{ type, target }]`. UI: в детальной карточке — кликабельные чипы связей по типам, клик открывает связанный инструмент. Позже — мини-граф.

## 13. Слой компетенций и грейдов (планируется)

Третий слой, связывающий инструменты с карьерным ростом. Источники — матрицы компетенций (см. справочные ссылки). Идея:

- **Компетенция ↔ технологии** (DevOps/CI-CD → Jenkins/GitLab/Docker; нагрузочное тестирование → КИП/Тест-центр).
- **Компетенция ↔ грейд** (Junior/Middle/Senior/…): что требуется на каждом уровне.
- Вид **«Грейд»**: выбрал уровень роли → видишь нужные компетенции и стоящие за ними технологии («что учить, чтобы вырасти»).

Открытые вопросы по матрицам: расшифровать легенду отметок `v`/`ж`/`Ж`; согласовать расходящиеся версии (листы «Аналитики» vs «Программисты»); встраивать отдельным слоем, не смешивая с фильтрами инструментов.

## 14. Экспертность, контрибьюторы и методология (планируется)

Ценность ландшафта — в мнении (зрелость, рекомендации, связи), а мнению нужна видимая экспертиза. Делаем ставку на коллективную экспертизу уважаемых в сообществе людей, а не одного автора.

**Согласие (обязательно, до публикации).** Фото, имя и контакты любого человека — только с явного разрешения. Ссылаться на публичные профили (Telegram, GitHub, Инфостарт), а не на личные контакты; аватар — только с согласия. Процесс: короткое согласование «как это будет выглядеть» + ссылка на черновик.

**Как показываем экспертность (вводим постепенно):**
1. **Экспертный совет** — небольшой список (3–5) именитых людей: фото, имя, роль/компания, ссылка на профиль, строка био; каждый ревьюит свою область.
2. **Контрибьюторы** — открытый список всех участников (как в awesome/CNCF), мотивирует вносить вклад.
3. **Атрибуция оценок** — «категорию курировал X», «метку зрелости подтвердили …» (приём Thoughtworks Tech Radar).

**Страница «Методология»** — как принимаются решения, кто участвует, как предложить правку. Прозрачность = доверие.

**Модель данных:** `experts: [{ name, role, photo, links, areas }]`; опционально `curatedBy` у категорий/элементов. Реализация — статикой.

**Порядок:** сначала доделать прототип (наполнение + связи) → согласовать 3–5 экспертов на черновике → добавить разделы «Эксперты» и «Методология» → открытые контрибьюторы.

## 15. Открытые вопросы

- **Логотипы**: для части инструментов нет официального SVG; на MVP — заглушки, источники определим позже.
- **Модель вклада** (CONTRIBUTING): как сообщество будет добавлять инструменты в ландшафт.
- **Порядок реализации**: сначала связи (п.12), затем слой компетенций (п.13), экспертность (п.14) — после наполнения прототипа.
